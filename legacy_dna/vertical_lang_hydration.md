# Vertical Language & Smart Hydration Engine (Legacy DNA)

## 1. جوهر الفكرة (The Concept)
في النظام القديم (`os/src`)، كانت هناك مشكلة شائعة في تصميم قواعد البيانات المتعددة اللغات: هل نضع اللغات كأعمدة في نفس الجدول (Horizontal) أم في جداول منفصلة (Vertical)؟
النظام اختار بعبقرية **التصميم الرأسي (Vertical Language)**. 

### كيف تعمل؟
1. **الجدول الأساسي (Base Table):** مثلاً `sbn_products`. يحتوي فقط على البيانات الثابتة غير القابلة للترجمة (مثل `price`, `created_at`, `status`).
2. **جدول الترجمة (Lang Table):** مثلاً `sbn_products_lang`. يحتوي على `product_id`، `lang`، والحقول النصية (`title`, `description`).

## 2. المحرك القديم (The Node.js Hydrator)
المشكلة في التصميم الرأسي هي صعوبة استرجاع البيانات للواجهة، حيث يتطلب الأمر الكثير من جمل الـ `JOIN`.
النظام القديم حل هذا عبر محرك ذكي جداً في الـ Backend:

### أجزاء المحرك القديم:
- **`smartSchema.js`:** يكتشف تلقائياً وجود جدول ينتهي بـ `_lang` مرتبط بجدول أساسي.
- **`i18nLoader.js`:** يقوم بـ (Flattening). يقرأ الجدول الأساسي، يجلب صفوف الترجمة، ويدمجها في كائن واحد في الذاكرة (Memory).
- **`hydrator.js` (السحر الحقيقي):** لا يكتفي بدمج الجدول الأساسي، بل يحلل العلاقات (Foreign Keys). فإذا كان المنتج ينتمي لقسم (Category)، يذهب الـ Hydrator لجلب بيانات القسم، ويجلب ترجمة القسم أيضاً، ويرسل للواجهة كائناً مسطحاً ومترجماً بالكامل:
  ```json
  {
    "id": "123",
    "price": 100,
    "category": { "id": "456", "name": "إلكترونيات" },
    "i18n": { "lang": { "ar": { "title": "هاتف" } } }
  }
  ```

هذا جعل الفرونت إند "غبياً" وسريعاً، لأنه يستلم البيانات جاهزة للعرض دون معالجة.

## 3. المشكلة في المحرك القديم
رغم قوة الفكرة، إلا أن تنفيذها في طبقة التطبيق (Node.js/JS) في وقت التشغيل (Runtime) كان يستهلك الكثير من الذاكرة والوقت لمعالجة الـ Arrays وربط العلاقات برمجياً.

## 4. التطور في النظام الجديد (AI-First Architecture)
للحفاظ على هذه العبقرية مع تحقيق أقصى أداء ممكن (C++ / DB level)، سنقوم بنقل مسؤولية الـ **Smart Hydration** من طبقة الكود إلى **طبقة قاعدة البيانات (Database Layer)** باستخدام **Dynamic SQL Views**.

### آلية العمل الجديدة (Database-Level Flatting):

بدلاً من معالجة البيانات برمجياً، سيقوم المحرك (Compiler) أثناء قراءة الـ Schema بتوليد `VIEW` مخصص لكل لغة، لكل جدول.

#### مثال على التوليد التلقائي (Auto-Generated Views):
عندما يقرأ الـ Compiler جدول `sbn_products` وجدول `sbn_products_lang`:
سيقوم تلقائياً بكتابة أمر الـ SQL التالي في قاعدة البيانات:

```sql
CREATE OR REPLACE VIEW view_sbn_products_ar AS
SELECT 
    p.*, 
    pl.title AS name,
    pl.description,
    cat.name AS category_name
FROM sbn_products p
LEFT JOIN sbn_products_lang pl 
    ON p.id = pl.product_id AND pl.lang = 'ar'
LEFT JOIN view_sbn_categories_ar cat 
    ON p.category_id = cat.id;
```

### فوائد هذه المعمارية الجديدة:
1. **أداء خارق (Zero-Cost Hydration):** قاعدة البيانات (PostgreSQL) أسرع بملايين المرات في دمج البيانات (JOIN) من أي لغة برمجة.
2. **تبسيط الـ C++ ORM:** لن يحتاج محرك C++ الجديد إلى بناء خوارزميات معقدة للدمج. هو فقط ينفذ `SELECT * FROM view_sbn_products_ar`.
3. **فرونت إند مطلق البساطة:** الواجهة تستلم JSON مسطحاً 100%، ولا تحتاج لأي منطق للتعامل مع الترجمات:
   ```json
   { "id": "123", "price": 100, "name": "هاتف", "category_name": "إلكترونيات" }
   ```
4. **دعم الـ UI Labels (Key-First):** ستُجمع نصوص الواجهة الثابتة (مثل "حفظ"، "تأكيد") من الجداول وتُحقن كـ `JSON` بصيغة `Key-First` عند بناء التطبيق، لتتولى النواة (MAS Core V2) معالجتها محلياً بسرعة.

## 5. الخلاصة الذهبية (Golden Rule for AI)
> **"الـ Backend يحتفظ بذكاء فصل اللغات رأسياً للحفاظ على النظافة القواعدية (Normalization)، لكنه يقدم البيانات مسطحة (Flattened) للواجهة عبر طبقة SQL Views لضمان سرعة الضوء في الأداء. الفرونت إند لا يعرف شيئاً عن تعدد اللغات سوى مفاتيح الـ UI."**
